🏗️ Full Architecture Diagram

Complete system architecture — Client → Gateway → Services → Data Layer → Observability.

📖 Architecture Explanation (Hindi)

Client Layer: Angular SPA Nginx pe serve hota hai (:8051). User browser se sab API requests API Gateway pe jaati hain.

Gateway Layer: JwtAuthGlobalFilter har request pe: (1) Token extract karta hai, (2) Signature validate karta hai, (3) Role check karta hai path ke against, (4) X-Authenticated-User, X-Authenticated-Role, X-Authenticated-UserId headers add karta hai, (5) Correct service pe route karta hai via Eureka lb://.

Infrastructure: Config Server sab services ko startup pe config deta hai. Eureka Server services ko register/discover karata hai.

Business Services: 8 domain services — Auth, User-KYC, Wallet, Transaction, Rewards, Admin, Notification, Integration. Har ek apna bounded context handle karta hai.

Data Layer: PostgreSQL (shared DB wallet_db, 20 tables), RabbitMQ (4 async queues), Redis (caching + idempotency + rate limiting).

Observability: Zipkin (distributed tracing via Micrometer Brave), ELK Stack (centralized logging), Prometheus+Grafana (metrics via /actuator/prometheus), SonarQube (code quality + JaCoCo coverage).

🔄 Internal Code Flow — Login Process

1

Request Received

AuthController.login(AuthRequest) → Calls AuthService.login()

2

Find User

repository.findByUsername() → If not found → throw "Invalid credentials"

3

Password Validation

Check if stored password is BCrypt hash (startsWith("$2")). If yes → passwordEncoder.matches(). If no (legacy plain-text) → direct string compare.

4

Auto-Migration

If password was plain-text and login successful → auto-hash with BCrypt and save. This ensures legacy users are migrated.

5

Generate JWT

jwtService.generateToken(user) → Token contains {userId, username, role, expiry}. Returns AuthResponse(token, role, "Login successful")

🛡️ Signup Role Validation

Users can only signup with roles: USER, SUPPORT, MERCHANT. ADMIN role cannot be self-assigned — prevents privilege escalation. Null/blank role defaults to USER.

Wallet Service — Deep Dive
Port: 8065 | Package: com.wallet.wallet | Patterns: CQRS, Idempotency, Event-Driven

Core financial service — balance management, topup, P2P transfer, payment gateway integration, daily limits, ledger.

📦 Entities (3 tables)

EntityKey Fields
WalletAccountid, userId, balance (BigDecimal)
LedgerEntryid, userId, type (TOPUP/TRANSFER_DEBIT/TRANSFER_CREDIT), amount, reference (UUID), createdAt
WalletLimitConfigid, dailyTopupLimit (₹50,000), dailyTransferLimit (₹25,000), dailyTransferCountLimit (10)

🔗 Feign + RabbitMQ

Feign Clients (Outgoing):

  • UserClient → User-KYC: Validate receiver exists for transfer
  • IntegrationClient → Integration: Init/Confirm payment gateway topup
  • NotificationClient → Notification: Send SMS+Email on topup/transfer

RabbitMQ Publishing:

  • wallet.events.queue → {userId, type, amount} — consumed by Transaction + Rewards
  • wallet.cqrs.events.queue → CQRS cache invalidation events
  • notification.queue → Notification events

🔄 Wallet Top-up Flow (Direct)

1

Validate Limits

Check daily topup sum from ledger (sumByUserAndTypeAndCreatedAtBetween). If exceeds ₹50,000 → reject.

2

Get or Create Account

walletAccountRepository.findByUserId(). If not found → create new with balance ₹0.

3

Update Balance

balance = balance + amount. Save to DB. @Transactional ensures atomicity.

4

Create Ledger Entry

New LedgerEntry(type=TOPUP, amount, reference=UUID). Immutable audit trail.

5

Publish Events

RabbitMQ: WalletEvent{userId, TOPUP, amount}. Notification: SMS + EMAIL to user.

🔄 P2P Transfer Flow

1

Validate Request

Sender ≠ Receiver. Both IDs required. Positive amount.

2

Check Limits

Daily transfer amount limit (₹25,000). Daily transfer count limit (10). Both checked from ledger.

3

Verify Receiver

Feign call → UserClient.getUserById(toUserId). If 404 → "Receiver user not found".

4

Balance Check

fromAccount.balance >= amount? If no → "Insufficient wallet balance".

5

Execute Transfer

Sender: balance -= amount. Receiver: balance += amount. Two ledger entries: TRANSFER_DEBIT + TRANSFER_CREDIT.

6

Events + Notifications

WalletEvent published. SMS to both sender ("Transfer successful") and receiver ("Amount received").

🔄 Payment Gateway Top-up Flow

1

Init Payment

POST /api/wallet/topup/init → Feign → IntegrationClient.initPayment() → Returns {paymentRef: "PAY-xxx", gatewayUrl}

2

User Pays Externally

User opens gateway URL, completes payment. Integration Service updates status to SUCCESS.

3

Confirm Payment

POST /api/wallet/topup/confirm/PAY-xxx → Feign → Check Integration status. If SUCCESS → topup wallet. If already CAPTURED → return existing. Mark as CAPTURED in Integration.

Port: 8068 | Package: com.wallet.transaction | Patterns: CQRS, Saga, Idempotency, Event-Driven

Most complex service — handles topup/transfer/payment/refund recording, Saga orchestration, dispute management, PDF receipts, CSV/PDF statements.

📦 Entities (3 tables)

EntityKey Fields
Transactionid, userId, senderId, receiverId, amount, type (TOPUP/TRANSFER/PAYMENT/REFUND), status (PENDING/SUCCESS/FAILED), idempotencyKey, createdAt
LedgerEntryid, transaction (FK), userId, entryType (DEBIT/CREDIT), amount
Disputeid, transactionId, userId, reason, status (OPEN/ESCALATED/RESOLVED), escalated, resolution

🔑 Idempotency Mechanism

Every request has an idempotencyKey. Before processing:

  • transactionRepository.findByIdempotencyKey(key)
  • If found → validate same payload → return existing (no duplicate)
  • If payload differs → throw IdempotencyConflictException
  • Isolation: @Transactional(isolation = Isolation.SERIALIZABLE)

🔄 Saga Pattern — Payment Flow (4 Steps + Compensation)

1

Step 1: CreateTransaction

Create Transaction(status=PENDING, type=PAYMENT) in DB.
⟲ Compensate: Mark transaction as FAILED

2

Step 2: ValidateAndReserve

Feign → UserClient.getUserById(senderId) + getUserById(receiverId). Check calculateBalanceForUpdate(senderId) >= amount.
⟲ Compensate: No-op (read-only)

3

Step 3: AccountingEntries

Create DEBIT ledger for sender + CREDIT ledger for receiver. Mark transaction SUCCESS.
⟲ Compensate: Reverse ledger entries

4

Step 4: PublishRewardEvent

Publish WalletEvent to RabbitMQ for reward points. Publish TransactionHistoryEvent for cache.
⟲ Compensate: Rewards rollback (logged)

❌ Failure: SagaOrchestrator.compensate() runs in REVERSE order — last completed step first. All exceptions logged but don't block other compensations.

🏆 Tier System

SILVER: 0 - 999 points (default)
GOLD: 1000+ points (configurable threshold)
PLATINUM: 5000+ points (configurable threshold)

Points Calculation: amount ÷ 100 × pointsPer100. E.g., ₹500 → 5 blocks × 1 = 5 points.

Tier auto-updates on every points change (earn or redeem).

🔄 Redeem Flow

1

Check Stock

item.getStock() > 0? If no → "Reward out of stock"

2

Check Points

account.points >= item.pointsCost? If no → "Insufficient points"

3

Deduct & Update

Points -= cost. Tier recalculated. Stock -= 1. RedeemCount += 1. RewardsTransaction(type=REDEEM) saved.

4

Notify

EMAIL + PUSH notification sent to user: "Reward redeemed: {item.name}"

🐰 Event-Driven Point Earning

WalletEventListener listens to wallet.events.queue. When wallet event received → calculateEarnPoints(amount) → addPoints(userId, points, "EARN"). Auto-updates tier.

API Gateway — Deep Dive
Port: 8062 | Spring Cloud Gateway | JwtAuthGlobalFilter

🔄 JwtAuthGlobalFilter — Request Processing Pipeline

1

Check Public Path

If path matches /api/auth/**, /swagger-ui/**, /v3/api-docs/**, /actuator/** → SKIP auth, pass through.

2

Extract Token

Read Authorization: Bearer {token} header. If missing → 401 Unauthorized.

3

Validate JWT

Parse token, verify signature with secret key, check expiry. If invalid → 401.

4

Role Authorization (RBAC)

Extract role from token. Match against path permissions:
• /api/admin/** → only ADMIN
• /api/support/** → only ADMIN, SUPPORT
• /api/merchant/** → only ADMIN, MERCHANT
• Other paths → all authenticated roles
If role not authorized → 403 Forbidden.

5

Mutate Request Headers

Add: X-Authenticated-User, X-Authenticated-Role, X-Authenticated-UserId. Remove original Authorization header. Forward to downstream service.

Section 13

🧩 Design Patterns Summary

CQRS

Wallet + Transaction: Separate Command/Query controllers. Writes → PostgreSQL. Reads → Redis cache. Cache invalidated via RabbitMQ events.

Saga (Orchestration)

Transaction Service: SagaOrchestrator<PaymentSagaContext>. 4 steps with compensation. Reverse-order rollback on failure.

Event-Driven

RabbitMQ: 4 queues, 6 listeners. Wallet events trigger Rewards+Transaction+Notification processing asynchronously.

API Gateway

Spring Cloud Gateway: Single entry, JWT auth, CORS, rate limiting, path-based routing via Eureka lb://.

Idempotency

Redis setIfAbsent + DB idempotencyKey check. Prevents duplicate financial operations.

Service Discovery

Netflix Eureka: Dynamic registration. No hardcoded URLs. Load-balanced routing.
