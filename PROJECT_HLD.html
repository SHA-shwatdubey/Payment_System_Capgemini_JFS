📖 Architecture Explanation

Client Layer: Angular frontend Nginx se serve hota hai port 8051 pe. Saare API calls API Gateway (:8062) pe jaati hain.

Gateway Layer: Spring Cloud Gateway har request pe JWT validate karta hai, role check karta hai, rate limit lagata hai, aur correct service pe route karta hai (Eureka lb:// se).

Infrastructure: Config Server sab services ko centralized config deta hai. Eureka Server services ko register karata hai for dynamic discovery.

Business Services: 8 microservices — har ek apna bounded context handle karta hai. Wallet aur Transaction me CQRS pattern hai. Transaction me Saga pattern bhi hai.

Data Layer: Ek shared PostgreSQL (wallet_db), RabbitMQ event bus (async communication), aur Redis (caching + idempotency).

Observability: Zipkin (tracing), ELK (logging), Prometheus+Grafana (metrics), SonarQube (code quality).

🔗 Inter-Service Communication

Services ek doosre se 2 tarike se baat karti hain — Synchronous (Feign Client) aur Asynchronous (RabbitMQ Events).

📡 Synchronous — OpenFeign Clients

Direct HTTP calls via Eureka load-balanced URLs. Jab immediate response chahiye tab use hota hai.

SourceTargetPurpose
💰 Wallet👤 User-KYCUser verify before wallet ops
💰 Wallet🔗 IntegrationExternal payment gateway topup
💸 Transaction👤 User-KYCValidate user before txn
👑 Admin👤 User-KYCKYC approval/rejection
👤 User-KYC🔗 IntegrationExternal KYC doc verification

🐰 Asynchronous — RabbitMQ Queues

Event-driven messaging. Fire-and-forget pattern. Services loosely coupled rehti hain.

QueuePublisher → Consumer
wallet.events.queueWallet → Transaction, Rewards
notification.queueWallet, Txn, KYC, Rewards → Notification
transaction.events.queueTransaction → Cache Invalidation
wallet.cqrs.events.queueWallet → Cache Invalidation

📖 Communication Pattern Explanation

Feign Clients: Jab ek service ko doosri service ka response immediately chahiye (e.g. Wallet ko User verify karna hai before topup), tab OpenFeign use hota hai. Eureka se service ka address milta hai (lb:// protocol).

RabbitMQ Events: Jab action ho gaya aur baaki services ko inform karna hai bina wait kiye (e.g. topup hua → rewards points do + notification bhejo), tab RabbitMQ event publish hota hai. Consumer services apne pace pe process karti hain. Isse loose coupling milta hai.

Section 04

🧩 Design Patterns — CQRS, Saga & More

Project me kaafi advanced design patterns use hue hain. Yahan detail me samjho har pattern kya karta hai aur kahan lagta hai.

🔄 CQRS — Command Query Responsibility Segregation

Used in: Wallet Service + Transaction Service

✍️ Command Side (Write)

  • WalletCommandController → /wallet/cmd/**
  • Writes directly to PostgreSQL
  • Uses Redis setIfAbsent for idempotency (duplicate prevention)
  • Publishes event to RabbitMQ after write
  • @CacheEvict clears stale cache

📖 Query Side (Read)

  • WalletQueryController → /wallet/query/**
  • Reads from Redis cache (@Cacheable)
  • On cache miss → reads from PostgreSQL
  • CacheInvalidationListener listens to RabbitMQ events
  • Evicts cache when write happens

✅ Benefit: Read operations bahut fast hain (Redis se) aur write operations consistent hain (DB se). Cache invalidation event-driven hai — eventually consistent model.

🔄 Saga Pattern — Orchestration Based

Used in: Transaction Service — Payment Flow

1

Validate User (Step 1)

Feign call to User-KYC Service → Check user exists & KYC approved

Compensate: No-op (read-only step)

2

Initiate Payment (Step 2)

Call Integration Service → Create payment record with PENDING status

Compensate: Refund payment via Integration Service

3

Record Transaction (Step 3)

Save Transaction entity in DB with status PROCESSING

Compensate: Delete transaction record from DB

4

Update Wallet (Step 4)

Debit/Credit wallet balance + Create ledger entry

Compensate: Reverse wallet balance change

❌ Agar koi step fail hota hai: SagaOrchestrator reverse order me sab completed steps ka compensate() call karta hai — rollback ho jaata hai automatically.

🔐 Security Architecture — JWT + RBAC

Poora security flow — login se lekar protected API access tak.

🔄 JWT Authentication Flow

1

User Login

POST /api/auth/login → Auth Service validates credentials (BCrypt) → Returns JWT token with {userId, username, role}

2

Frontend Stores Token

Angular stores JWT in localStorage. Har subsequent request me Authorization: Bearer <token> header lagta hai.

3

Gateway Validates

JwtAuthGlobalFilter token validate karta hai, role extract karta hai, path authorization check karta hai.

4

Headers Forwarded

Gateway request me X-Authenticated-User, X-Authenticated-Role, X-Authenticated-UserId headers add karta hai.

5

Service Reads Headers

Downstream services in headers se user identity padhti hain. No re-validation needed.

Section 07

💵 End-to-End Payment Flow

User jab "Add Money" karta hai tab paise kaise move hote hain — complete sequence.

🔄 Wallet Top-up via Payment Gateway — Step by Step

1

User clicks "Add Money ₹500"

Frontend → POST /api/wallet/topup/init {amount: 500} → API Gateway

2

Wallet → Integration Service (Feign)

Wallet calls Integration Service via Feign: POST /api/integrations/payments/init
Integration Service creates PaymentTransaction(status=PENDING) in DB, returns {paymentRef: "PAY-xxx"}

3

User completes payment

Frontend shows payment page. User confirms. Then calls POST /api/wallet/topup/confirm/PAY-xxx

4

Wallet verifies + updates balance

Wallet → Feign call to Integration → Check payment status = COMPLETED
Wallet updates: balance += 500, creates LedgerEntry

5

Events Published to RabbitMQ

wallet.events.queue → {userId, amount, type=TOPUP}
notification.queue → {userId, "Top-up of ₹500 successful"}

6

Async Processing

🎁 Rewards Service listens → Awards 5 reward points
💸 Transaction Service listens → Records transaction log
🔔 Notification Service listens → Sends email/SMS

☁️ Deployment Architecture — AWS + Docker + Terraform

Production deployment — Terraform se AWS EC2 provision hota hai, Docker Compose se sab containers chalte hain.

🏗️ Terraform Infrastructure

  • Provider: AWS (~5.0)
  • EC2: Ubuntu 22.04, gp3 EBS volume
  • SSH: Auto-generated RSA 4096 key pair
  • Security Group: Ports 22, 5432, 8051-8070, 15672
  • User Data: Auto-installs Docker + deploys compose

🐳 Docker Setup

  • 22 Containers in single docker-compose.yml
  • Custom Dockerfiles for each service (multi-stage build)
  • Docker Hub: shashwattech/capgemini_javafullstack_done
  • Network: Bridge network microservices
  • Volumes: Persistent data for PG, RabbitMQ, Grafana, Prometheus
  • Health checks on all services

🔄 Service Startup Order

1️⃣ PostgreSQL + RabbitMQ + Redis + Zipkin

⏱️ depends_on with condition: service_healthy ensures correct ordering. Config Server must be healthy before business services start.
